# 1  The two-paragraph elevator pitch

QubiC is a superconducting-qubit control system built at LBNL, running
on the Xilinx ZCU216 RFSoC evaluation board. For every gate the system
plays — an X90 rotation that turns a qubit by 90°, a readout tone that
measures it, a marker pulse that tells other equipment "go" — the FPGA
needs to push a complex *envelope waveform* into the digital-to-analog
converter (DAC) at one billion samples per second. Today those
envelopes live in on-chip block RAM as raw samples: one signed 16-bit
$I$ (in-phase) value and one $Q$ (quadrature) value per nanosecond. A
2 µs readout pulse therefore costs $2000 \cdot 4 = 8\,$kB of memory;
multiply by the number of qubits and gates and the envelope memory
dominates the gateware's storage budget. Worse, every time a
calibration routine nudges a pulse parameter, the new sample buffer
has to be re-uploaded over PCIe — which is slow.

**QubiCSpline replaces those raw samples with a cubic spline that the
FPGA evaluates on the fly.** Instead of $n$ samples per pulse, we
store $k \ll n$ *knots* plus four polynomial coefficients per segment;
at runtime, a small piece of SystemVerilog in
[rtl/spline_eval.sv](../rtl/spline_eval.sv) computes one output sample
per clock from those coefficients. For our typical readout pulse this
drops storage by **48×**; for neutral-atom transport waveforms (the
Lukin-group use case) by **~2700×**. Crucially, we preserve the exact
35-clock-cycle latency of the existing QubiC envelope path, so the
new module is a drop-in replacement: no changes required to
`pulse_reg.sv`, `pulse_iface.sv`, `proc.sv`, or the QubiC control
software.

By the end of this document, you should understand what every file in
the project does, why it exists, the math behind the algorithm, the
hardware constraints that shaped the design, the test suite that
proves correctness, and the Vivado synthesis result that says the
hardware is real.

# 2  Background, kept just rigorous enough

## 2.1  Primer: what an "envelope" is

> **Primer — Pulse envelopes for quantum control.** A qubit is steered
> by an RF tone (typically 4–8 GHz). The frequency of the tone selects
> *which* qubit; the *envelope* — a slow amplitude modulation on top
> of the carrier — controls *how* the qubit rotates. A square envelope
> with sharp edges drives a fast, imprecise rotation. A Gaussian
> envelope drives a slower, much cleaner rotation. A *DRAG* envelope
> (Gaussian + a small derivative correction on the quadrature
> component) cancels unintended leakage to the third energy level
> $|f\rangle$. The envelope is what the FPGA generates; the carrier is
> generated by a numerically controlled oscillator (NCO) and mixed
> with the envelope before going to the DAC.

QubiC sends each envelope sample as a signed 16-bit $\{I, Q\}$ pair at
1 GSPS (giga-samples per second). $I$ is the in-phase component and
$Q$ is the quadrature (90°-shifted) component. Together they specify
a complex amplitude.

## 2.2  Primer: what an FPGA is, and the pieces we use

> **Primer — FPGA fabric.** An FPGA is a chip whose internal logic is
> defined by configuration data loaded at power-on (the *bitstream*).
> Our chip is the Xilinx Zynq UltraScale+ RFSoC `xczu48dr` on the
> ZCU216 evaluation board. Inside, the relevant building blocks are:
>
> - **LUTs** (look-up tables) — programmable boolean logic. A 6-LUT
>   computes any function of 6 inputs.
> - **Flip-flops (FFs)** — single-bit registers that capture a value
>   on each clock edge. State lives here.
> - **DSP slices (DSP48E2)** — dedicated multiplier-adder blocks.
>   Each can do a signed 27×18-bit multiply with an accumulator. Far
>   faster and more power-efficient than building a multiplier out of
>   LUTs.
> - **Block RAM (BRAM)** — 36-kilobit on-chip memory tiles. Each tile
>   can be configured as one 36-bit-wide port or two 18-bit ports.
>   The two BRAMs we use (coefficient and width memory) hold thousands
>   of spline segments.
> - **Clock buffers** — drive a clock net across the whole chip with
>   skew under a nanosecond.
>
> The chip has **425 280 LUTs, 850 560 FFs, 4272 DSP slices, 1080 BRAM
> tiles** available. Our entire spline evaluator uses **258 LUTs, 391
> FFs, 7 DSPs, 0 BRAM tiles** — under 0.2% of any resource. Splines
> are extremely cheap.

The "0 BRAM tiles" looks wrong at first glance — we definitely use
BRAM. The reason it shows zero in our utilization report is that the
two BRAMs are *external* to the `spline_eval` module: they sit on the
other side of its port boundary and get instantiated separately as
Vivado Block Memory Generator IP. When we synthesize `spline_eval`
alone (for this design-rule check), the BRAMs don't count.

## 2.3  Cubic splines, in one figure of merit

A **cubic spline** is a piecewise polynomial of degree 3 defined on an
interval partitioned by *knots* $t_0 < t_1 < \dots < t_n$. On each
segment $[t_i, t_{i+1}]$ the spline takes the form
$$
S_i(t) \;=\; a_i + b_i(t-t_i) + c_i(t-t_i)^2 + d_i(t-t_i)^3
$$
with the coefficients chosen so the resulting curve and its first
two derivatives are continuous across knots — *$C^2$ continuity*.
This is enough smoothness that the spline cannot create a discontinuity
in the signal or its slope or its curvature. The standard
*not-a-knot* boundary condition (what `scipy.interpolate.CubicSpline`
uses by default) forces the third derivative to be continuous at the
second and second-to-last knot too, which avoids end-effects that
"natural" splines suffer from. We use not-a-knot throughout.

**Why cubic splines, specifically?** A few alternatives we considered
and didn't pick:

| Alternative                     | Why not                                                                |
|---------------------------------|-------------------------------------------------------------------------|
| Just denser raw samples         | Defeats the whole point — that's what we're replacing.                  |
| Linear interpolation            | Not smooth enough; corners would distort the DRAG derivative response.  |
| Quadratic splines               | $C^1$ only — slope discontinuities at knots would inject high frequency.|
| Higher-degree polynomials       | More coefficients per segment $\Rightarrow$ less storage win, more DSPs.|
| Chebyshev / Fourier expansions  | Globally non-local; one knot moves and the whole pulse shifts.          |
| Bézier curves                   | Mathematically equivalent for cubics; just different basis. No benefit. |

Cubics are the sweet spot: smooth enough that the DAC can't tell the
difference from "infinite knot density", cheap enough that one segment
is just four 16-bit numbers, and *local* — each segment only cares
about its own knots.

## 2.4  Horner evaluation

We evaluate the polynomial in **Horner form**:
$$
S(u) \;=\; a + u\bigl(b + u\bigl(c + u\,d\bigr)\bigr)
$$
This is **three multiplications and three additions** instead of the
six multiplies and three adds of the naive form. Each multiplication
is one DSP slice. Per channel ($I$ and $Q$) we need 3 DSPs; plus one
more for the $u$ normalization (Section 2.6); so **7 DSPs total** —
the number that ended up in our synthesis report.

> **Primer — Horner.** Try it on paper. For $S(u) = a + bu + cu^2 +
> du^3$, you'd naively compute $u^2 = u \cdot u$, then $u^3 = u^2
> \cdot u$, then $b \cdot u + c \cdot u^2 + d \cdot u^3 + a$ — six
> multiplies. Horner factors it as $a + u(b + u(c + u \cdot d))$:
> compute innermost first, work outward. Three multiplies. Same
> answer, half the hardware.

The variable $u$ is a *normalized intra-segment time*: at the start of
segment $i$, $u = 0$; at the end, $u \to 1$.

## 2.5  Q1.15 fixed-point, and the DAC LSB

> **Primer — Fixed-point arithmetic.** Floating-point numbers are
> expensive in hardware (an IEEE 754 multiplier is many times bigger
> than an integer multiplier). For DSP work, *fixed-point* is the
> standard alternative: an integer with a notional binary point at a
> fixed position. "Q1.15" means: 1 sign bit, no integer bits, 15
> fraction bits. So the format represents numbers in the range
> $[-1, +1 - 2^{-15})$ with precision $2^{-15} \approx 3.05 \times
> 10^{-5}$.

Why Q1.15 and not something else?

- The QubiC `amp` field is always a DAC fraction in $[0, 1)$.
  Q1.15 ±1.0 covers it.
- The DAC is 16-bit, so 1 Q1.15 LSB equals **1 DAC LSB** — the
  hardware's own resolution floor. We can't usefully be more precise
  than the DAC anyway.
- It fits cleanly in a signed 16-bit register, which is one of the
  natural widths in the DSP48E2 (18-bit signed).
- Q1.15 × Q1.15 = Q2.30 in 32 bits; the result's bits [30:15] are the
  Q1.15 product. Clean, no rounding decisions hidden in the math.

The project's **error budget is 4 DAC LSB** — meaning the spline
evaluator may not produce any sample more than 4 LSB away from the
ideal float waveform. This is *well* below qubit gate-fidelity
noise: the dominant gate-error source is qubit decoherence, around
$10^{-3}$ relative error, while 4 LSB on a typical amp of 0.2 is
$4 \cdot 2^{-15} / 0.2 \approx 6 \times 10^{-4}$. So even at the
budget ceiling we're not the limiting source of error in the gate.

## 2.6  How $u$ is computed in hardware

The hardware doesn't know about wall-clock time; it counts in clock
cycles. Inside segment $i$, a counter `u_cnt` increments every cycle
from 0 to $h_i - 1$ (where $h_i$ is the segment width in clock
cycles). The normalized $u$ for the Horner pipeline is
$$
u \;=\; \frac{u_\mathrm{cnt}}{h_i}
\;=\; \bigl(u_\mathrm{cnt} \cdot \mathrm{recip\_h}\bigr) \gg 15
$$
where $\mathrm{recip\_h} = \mathrm{round}(2^{15} / h_i)$ is precomputed
in Python and stored in the width BRAM alongside $h_i$. **This avoids
a hardware divider** (dividers are expensive — multi-cycle and large).
One DSP multiply replaces the division.

Coefficients are also pre-scaled in Python so the hardware sees
$u \in [0, 1)$ rather than $t - t_i \in [0, h_i)$. Specifically:
$$
a_\mathrm{hw} = a,\quad b_\mathrm{hw} = b \cdot h_s,\quad
c_\mathrm{hw} = c \cdot h_s^2,\quad d_\mathrm{hw} = d \cdot h_s^3
$$
where $h_s$ is the segment width in seconds. This keeps all
coefficient magnitudes in $(-1, 1)$ for typical QubiC amplitudes,
which fits Q1.15.

**Why pre-scale in Python instead of hardware?** Two reasons:

1. The compile step happens once per calibration update (~10 ms);
   the hardware runs at 1 GSPS forever. Move expensive math to the
   side where it's cheap.
2. If hardware did the scaling, it would need to know $h_s$ in
   seconds, not just $h_i$ in cycles. Now the hardware never sees
   physical time — just integer cycle counts.

## 2.7  AutoKnots, in one figure of merit

Given an envelope function $f(t)$ on an interval and a relative
tolerance $\delta$ (we use $10^{-3}$ by default), the **AutoKnots**
algorithm (Vitenti et al.,
[arXiv:2412.13423](https://arxiv.org/abs/2412.13423)) returns the
*smallest* knot set such that the cubic spline $\hat f$ through those
knots satisfies, on every interval $[t_i, t_{i+1}]$:

1. **Pointwise condition** —
   $|f(t_m) - \hat f(t_m)| \;<\; \delta\,(|f(t_m)| + \varepsilon)$,
   where $t_m$ is the segment midpoint.

2. **Integral condition** —
   $\big|\!\int_{t_i}^{t_{i+1}} f - \int_{t_i}^{t_{i+1}} \hat f\big|
   \;<\; \delta\,(|\!\int f| + \varepsilon \cdot h_i)$.

The first condition catches obvious mismatches; the second catches
*hidden* features that pointwise midpoint sampling would miss
(imagine a thin spike straddling a wide segment — the midpoint test
gives it a pass, but the integral test won't).

The algorithm starts with $n_\text{init}$ uniform knots, then
**bisects** any interval failing either check, refits, and iterates
until everything passes. This is greedy and converges in $O(\log)$
iterations for smooth signals.

> **Primer — why this matters for compression.** A flat region of a
> pulse needs almost no knots; a Gaussian shoulder needs many. A
> uniform sampling rate is forced to oversample the flat to hit the
> shoulder's accuracy. Adaptive knot placement matches knot density
> to local curvature, giving compression ratios that scale with the
> *information* in the pulse, not its duration.

The published algorithm doesn't know what hardware it's targeting.
Three extensions make it produce hardware-shippable splines — see
Section 6.

# 3  Architecture: two halves that meet at a BRAM

The whole project cleanly splits into **offline Python** (runs on the
control workstation) and **online FPGA** (runs in the bitstream at
250 MHz). The two halves only ever exchange BRAM contents and a few
small control words.

```
┌──────────────────────────── OFFLINE (laptop, Python) ───────────────────────────┐
│                                                                                  │
│   qubitcfg.json ─► pack_qubitcfg.py ─► for each gate:                            │
│                                            compile_pulse()                       │
│                                                │                                 │
│                                                ▼                                 │
│                                        make_envelope()                           │
│                                          I_func, Q_func                          │
│                                                │                                 │
│                                                ▼                                 │
│                                  compress_multichannel()  ──► autoknots()        │
│                                       (unified knot grid)                        │
│                                                │                                 │
│                                                ▼                                 │
│                                       pack_coefficients()                        │
│                                  (CubicSpline → Q1.15 BRAM words)                │
│                                                │                                 │
│                                                ▼                                 │
│                            spline_coeff.{mem,coe}, spline_width.{mem,coe},       │
│                            spline_manifest.json                                  │
└──────────────────────────────────────────────────────────────────────────────────┘
                                       │
                                       │  bitstream init  /  runtime BRAM writes
                                       ▼
┌────────────────────────── ONLINE (ZCU216 fabric, 250 MHz) ──────────────────────┐
│                                                                                  │
│   cmdstb ──► spline_eval.sv  ──► envdata_out  ──► (existing QubiC ammod path)    │
│                  │       ▲                                                       │
│                  ▼       │                                                       │
│            segment FSM   │  coeff_data[127:0] from coeff_bram (128b × 4096)      │
│            u_cnt counter │  width_data[31:0]  from width_bram (32b  × 4096)      │
│                  │                                                               │
│                  ▼                                                               │
│         4-stage Horner + Stage 1a + saturating output + 27-cycle delay           │
│                                                                                  │
└──────────────────────────────────────────────────────────────────────────────────┘
```

**The 35-cycle latency contract.** The right-hand side has a hard
constraint: 35 clock cycles from `cmdstb` (a one-cycle pulse meaning
"start the gate") to the first valid sample at `envdata_out`. This
number is baked into the rest of the QubiC gateware — `proc.sv` and
`pulse_iface.sv` schedule events assuming exactly this latency. We
cannot change it without touching many other files. So
`spline_eval.sv` is built around preserving it: pad whatever pipeline
depth the Horner evaluation needs (7 cycles originally, 8 after our
recent fix) with a `reg_delay1` shift register of length `OUT_DELAY`
so that the total comes out to 35.

This sounds like wasted cycles — and it would be, if we were free to
redesign the surrounding gateware. We aren't. **Trading 27 cycles of
shift-register depth for zero edits to four other gateware files is
correct engineering for this project.** The shift register costs a
handful of fabric flops; the alternative costs an integration
campaign.

## 3.1  Why split the work this way?

Three reasons the offline/online split is so clean:

1. **Calibration cadence.** Pulse parameters change during
   calibration sweeps (~once per second per parameter). Recompiling
   a spline at 10 ms per pulse is fast enough to live in the
   calibration loop. Running AutoKnots on the FPGA would be
   ridiculous: it's an iterative algorithm with control flow that
   doesn't fit cleanly in fabric, and there's no benefit because the
   recompile cadence is human-slow.

2. **Mathematical surface area.** All the floating-point math —
   Simpson's rule, cubic-spline construction, bisection — stays in
   Python where `scipy` does it for us. Hardware sees only signed
   16-bit integers and a 7-DSP pipeline.

3. **Verification.** The Python compiler is testable in plain Python
   (fast, no simulator needed). The hardware is testable with cocotb
   driving the SystemVerilog (cycle-exact). Each side has its own
   test surface; bugs are attributable to one side or the other.

# 4  The RTL

There are only two SystemVerilog files. One ships in the bitstream;
the other is a testbench harness.

## 4.1  `rtl/spline_eval.sv` — the synthesizable evaluator

This is the only file that ends up in hardware. Module declaration
(some comments elided):

```systemverilog
module spline_eval #(
    parameter SEG_ADDRW    = 12,   // BRAM depth = 2^12 = 4096 segments
    parameter OUT_DELAY    = 27,   // latency-padding shift register depth
    parameter MIN_H_CYCLES = 4     // runtime assertion floor
)(
    input  wire                  clk,
    input  wire                  cmdstb,      // 1-cycle pulse → start gate
    input  wire                  reset,
    input  wire [SEG_ADDRW-1:0]  coeff_base,  // BRAM start address
    input  wire [SEG_ADDRW-1:0]  n_segs,      // # of segments to play
    output reg  [SEG_ADDRW-1:0]  coeff_addr,  // → external coeff BRAM
    input  wire [127:0]          coeff_data,
    output reg  [SEG_ADDRW-1:0]  width_addr,  // → external width BRAM
    input  wire [31:0]           width_data,
    output wire [31:0]           envdata_out, // {I[15:0], Q[15:0]}
    output wire                  valid_out,
    output wire                  busy_out
);
```

The two BRAMs are *external* — the module just outputs addresses and
consumes the returned data. This matches how QubiC already organizes
envelope memory and means runtime BRAM updates (for calibration) still
work via the existing path.

### 4.1.1  Segment controller (lines 124–167)

A small finite state machine (FSM). On `cmdstb` it arms (`active <=
1`) and zeros the segment counter and the intra-segment cycle counter
(`u_cnt`). On each subsequent cycle:

- `u_cnt` increments until it hits `h_cycles - 1` (the segment's last
  cycle).
- On the last cycle, `u_cnt` zeroes and either `seg_idx` increments
  (more segments to play) or `active` drops (pulse done).
- Every cycle while active, `h_cycles` and `recip_h` are latched from
  the width BRAM port.

### 4.1.2  Address prefetch (lines 184–213)

> **Primer — why prefetch.** The BRAM has *read latency 2*: an
> address registered at clock $T$ gives data at $T+2$. If we want
> segment $i+1$'s data at the first cycle of segment $i+1$, the
> address has to be issued earlier. With one additional cycle for the
> address-register output, that's **3 cycles early**. So the FSM
> watches for `u_cnt == h_cycles - 3` and issues the next-segment
> address at that moment.

There's a subtle bug-trap here: once the prefetch fires, the
comparator goes false the next cycle, and naively the address would
revert to the current segment — wrong data. A `prefetched` register
captures the fire and holds the next-segment address until the
segment actually advances. This kind of bug is invisible if every
test pulse has wide segments ($h_\text{cycles} > 5$). It's exactly
why the adversarial constraint tests in Section 7 exist.

### 4.1.3  Q1.15 multiplier (lines 217–233)

A SystemVerilog `function` computing `q115_mul(a, b) = (a * b)[30:15]`
— the Q1.15 product. Synthesizes to **one DSP48E2 per call site**.
Used 6 times in the Horner stages (3 per channel) plus 1 more for the
$u$ normalization → 7 DSPs total. This is exactly the count we see in
the synthesis utilization report.

### 4.1.4  Horner pipeline (lines 235–363)

> **Primer — pipelining.** A combinational circuit (no flip-flops in
> between) is bounded by the speed of light through silicon: the
> longer the logic chain, the slower the maximum clock frequency.
> *Pipelining* breaks a long chain into stages separated by
> flip-flops. Each stage can be shorter and therefore faster; the
> total throughput is one result per clock at the new (higher) rate,
> at the cost of *latency* (results emerge a few cycles later).

The original design had 4 Horner stages:

| Stage | Computation                                  |
|-------|----------------------------------------------|
| 1     | latch $a, b, c$ from BRAM + compute $p_1 = d \cdot u$ |
| 2     | $p_2 = (p_1 + c) \cdot u$                    |
| 3     | $p_3 = (p_2 + b) \cdot u$                    |
| 4     | $out = p_3 + a$, with saturation             |

In Section 8 we added a **Stage 1a** in front of Stage 1 — a register
that captures $u$ and the raw `coeff_data` so the two
back-to-back DSP multiplies (the $u$ normalization and Stage 1's
$d \cdot u$) don't have to fit in one clock period. That's the
defensive-engineering fix; details in Section 8.

The key correctness rule: the *same* $u$ value must be used at all
four Horner stages. A different $u$ per stage would compute a
different polynomial. The `s1_u`, `s2_u`, `s3_u` registers propagate
stage 1's $u$ forward.

### 4.1.5  Saturation (lines 334–351)

> **Primer — overflow vs. saturation.** A 16-bit signed addition that
> overflows wraps to a large negative number (two's complement). For
> an audio sample or a DAC value, this is catastrophic: a tiny
> overshoot at the top of a pulse becomes a wildly negative spike at
> the bottom. *Saturation* replaces the wrapping behavior with a clamp
> — if the sum exceeds $+(2^{15} - 1)$, output $+(2^{15} - 1)$
> instead. The DAC sees a flat top instead of a sign-flip.

The final stage adds $a$ to $p_3$ with explicit saturation logic.
Without this, spline overshoot near the flat top of a near-unity-amp
pulse would produce DAC glitches.

### 4.1.6  Output delay (lines 365–377)

Two `reg_delay1` shift registers (one for the 32-bit `envdata`, one
for `valid_out`) of depth `OUT_DELAY = 27`. Pads the 8-cycle Horner
pipeline up to exactly 35 cycles total. The chains are mapped to SRL
primitives — efficient packed shift registers inside CLBs.

### 4.1.7  Simulation-only assertion (lines 174–182)

Wrapped in `// synthesis translate_off ... translate_on`, so it
disappears from the bitstream but fires in simulation. If a stale
BRAM image with `h_cycles < MIN_H_CYCLES` is loaded, the prefetch
math wraps and produces gibberish — but the wrap is silent, the FSM
runs, the output looks "plausibly wrong". The assertion catches this
class of bug at the first cycle instead of letting it propagate
through the whole pulse.

## 4.2  `rtl/spline_eval_top.sv` — the simulation harness

A non-synthesizable wrapper for cocotb testing. Defines:

- `reg_delay1` — the QubiC shift-register primitive used inside
  `spline_eval`. This lives here because the harness needs it; the
  real QubiC gateware library would supply it for production.
- `spline_eval_top` — a wrapper that owns two SystemVerilog memory
  arrays (`coeff_bram[]`, `width_bram[]`) modeling the external
  BRAMs, plus the DUT itself.

Cocotb populates the memory arrays directly from Python
(`dut.coeff_bram[i].value = ...`) before firing `cmdstb`. The
harness module is *not* synthesized: in
[scripts/synth_check.tcl](../scripts/synth_check.tcl), both files are
given to Vivado but the `-top spline_eval` flag tells the elaborator
to walk only from `spline_eval`. The harness module ends up as dead
code from the tool's point of view.

# 5  The Python pipeline

Six modules plus three test files plus three plotting scripts. The
dependency graph is small:

```
pack_qubitcfg.py        ←── top-level driver (qubitcfg.json → BRAM images)
       │
       ▼
spline_pulse_compiler.py    ←── single-pulse compile: paradict → BRAM words
       │
       ├─► autoknots.py            ←── adaptive knot allocation (the algorithm)
       ├─► pulse_envelopes.py      ←── DRAG, cos_edge_square, atom_transport, ...
       └─► spline_coeff_pack.py    ←── CubicSpline → Q1.15 words + .mem/.coe writer
```

## 5.1  `python/autoknots.py`

Implementation of the AutoKnots algorithm with the three hardware
extensions. Two public entry points:

- **`autoknots(f_func, t_start, t_end, ...)`** — fit one channel.
  Returns a result dataclass with the spline, knots, convergence
  flag, and a history log.
- **`compress_multichannel(f_funcs, ...)`** — fit several channels
  separately and *union* their knot sets, then refit each channel on
  the unified grid. We use this so $I$ and $Q$ share one segment-width
  BRAM (you can't have one BRAM holding two different segmentations of
  the same time axis).

The interesting internal function is `_loop` (lines 209–323). Each
iteration: build a `CubicSpline` through the current knots; evaluate
both convergence conditions on every interval; produce a boolean mask
of intervals needing subdivision; intersect that mask with the three
hardware constraints (Section 6); split or freeze accordingly.

`spline_integrals` (line 30) is the closed-form per-segment integral
of a cubic — the LHS of the integral condition. Because it's closed
form (no numerical quadrature error), the convergence guarantee is
clean.

`_snap` (line 199) is the no-op-when-disabled wrapper that snaps a
knot proposal to the `grid_quantum` grid. Critically, **snapping
happens before the error check**, not after. This means the
algorithm's convergence guarantee applies to the on-grid spline that
the hardware actually executes, not a slightly-different float spline
that would silently shift in the packer.

## 5.2  `python/pulse_envelopes.py`

One Python function per `env_func` name seen in `qubitcfg.json`.
Each returns a pair of callables `(I_func, Q_func)` that take a
NumPy time array.

| `env_func`        | $I$ channel                                  | $Q$ channel                               |
|-------------------|----------------------------------------------|-------------------------------------------|
| `DRAG`            | Gaussian                                     | scaled derivative (anti-leakage correction)|
| `cos_edge_square` | raised-cosine-edge square (readout)          | zero                                      |
| `square`          | constant                                     | zero                                      |
| `mark`            | constant (marker pulse)                      | zero                                      |
| `adiabatic_ramp`  | minimum-jerk ramp + hold (atom shuttling)    | zero                                      |
| `rydberg_drag`    | Gaussian                                     | **linear chirp (detuning, not quadrature)** |
| `atom_transport`  | minimum-jerk position polynomial             | zero                                      |

The "channel 2 semantic shift" callout at the top of the file matters:
for `rydberg_drag`, channel 2 is *detuning*, not quadrature. The
spline engine is channel-agnostic; downstream gateware has to know
what each channel means per envelope.

The module also exports `feature_knots(env_func, ...)`, which returns
*interior time points the spline fitter MUST place a knot at* — e.g.
the ramp/hold junction in `adiabatic_ramp` is a $C^2$ discontinuity
(the ramp's third derivative jumps at the junction), and the
algorithm's midpoint test would otherwise miss it and produce a wide
segment straddling the kink.

## 5.3  `python/spline_coeff_pack.py`

`scipy` CubicSpline → BRAM words. Three steps inside
`pack_coefficients`:

**Step 1: Validate `h_cycles`** against `MIN_H_CYCLES = 4` and
`MAX_H_CYCLES = 65535`. Both are hard errors at this layer. The
AutoKnots algorithm is supposed to enforce both via `min_dt` and
`max_dt`, but defense-in-depth: if anyone bypasses the algorithm and
hands us bad splines, we refuse to pack them silently.

**Step 2: Pre-check Q1.15 overflow on $d$.** The $d$ coefficient is
scaled by $h_s^3$ (Section 2.6), so for long pulses with wide
segments $d_\mathrm{hw} = d \cdot h_s^3$ can exceed $\pm 1.0$ even
when the underlying signal is bounded. Catching this here gives a
multi-line diagnostic listing the offending segments; without it, the
to-Q1.15 conversion would silently clip and produce a slightly wrong
waveform.

**Step 3: Per-segment Q1.15 packing.** For each segment: compute
$h_s$, $h_\mathrm{cyc}$, $\mathrm{recip}$; pull $a, b, c, d$ from
scipy's pp-form; scale by powers of $h_s$; round to Q1.15; pack into
the 128-bit coefficient word and 32-bit width word per the layout in
the RTL header.

Three I/O helpers (`write_mem`, `write_coe`, `verify`) write the BRAM
images and run a float simulation of the Q1.15 Horner pipeline.

## 5.4  `python/spline_pulse_compiler.py`

The single-pulse entry point.
`compile_pulse(env_func, paradict, twidth, amp)` returns a
`CompiledPulse` dataclass with `coeff_words`, `width_words`, knots,
segment count, convergence flag, raw-sample-equivalent count.

The body is straightforward:

1. Look up `delta` (per-shape default in `_DEFAULT_DELTA`, falling
   back to $10^{-3}$).
2. `make_envelope` to build $I_\text{func}, Q_\text{func}$.
3. Compute `min_dt = MIN_H_CYCLES / fs`, `max_dt = MAX_H_CYCLES / fs`,
   `grid_quantum = 1/fs`.
4. `feature_knots` to get any junction-seed knots.
5. `compress_multichannel` to fit both channels on a unified grid.
6. `pack_coefficients` to produce BRAM words.

The `_DEFAULT_DELTA` table is heavily annotated. Every per-shape
entry has a measured `(delta, n_segs, compression, max_err_lsb)`
table justifying the chosen value:

> Rule: pick the **loosest** delta for which `max_err_I_lsb < 4`
> holds at representative parameters. Tighter wastes BRAM; looser
> fails the test.

This isn't "I tried this and it worked" — it's "I measured the
delta-sweep, here are the receipts, here's the cheapest one that
meets budget." Anyone tuning a new shape later can see exactly what
ground was already covered.

The companion `verify_pulse(cp)` re-runs the Q1.15 Horner pipeline
in float (not in simulation) and returns max/RMS error in DAC LSBs.
Cheap, fast, runs without a simulator.

## 5.5  `python/pack_qubitcfg.py`

End-to-end driver. Reads `config/qubitcfg.json`, calls `compile_pulse`
on every gate's envelope, concatenates the resulting coefficient
words into a single BRAM image, allocates a base address per gate,
and writes:

- `build/spline_coeff.mem`, `build/spline_coeff.coe` (128b × 4096)
- `build/spline_width.mem`, `build/spline_width.coe` (32b × 4096)
- `build/spline_manifest.json` (per-gate base, n_segs, compression,
  error)

The `--only-compression-wins` flag skips gates where the spline
representation is *larger* than the raw-sample equivalent (typically
short DRAG X90 pulses at 32 ns that hit the `min_dt` floor too hard
to benefit). Those gates fall through to the existing raw-sample
path. The two paths can coexist — they just use different BRAMs.

# 6  The three hardware-aware constraints

The published AutoKnots algorithm doesn't know what hardware it's
targeting. Three extensions make it produce splines that are
guaranteed to round-trip through Q1.15 packing and FPGA evaluation
without surprises.

## 6.1  `min_dt` — minimum knot spacing

**Physical origin: the time DAC.** The hardware's time axis is
quantized: a counter increments once per clock cycle. The
prefetch FSM compares `u_cnt == h_cycles - 3`; if `h_cycles < 3`,
this comparator silently wraps and the FSM goes off the rails. We
add one cycle of safety margin and require **`MIN_H_CYCLES = 4`**.

**Behavior:** intervals narrower than $2 \cdot \mathrm{min\_dt}$ are
*frozen* — the algorithm warns once via `RuntimeWarning` and returns
the best fit achievable at that resolution. Without this, a
sufficiently steep pulse (a Gaussian spike with $\sigma$ smaller than
a few ns) drives the bisection chain into infinite depth.

**Pinned in three places** that must stay in sync:

| Location                                                     | Symbol                       |
|--------------------------------------------------------------|------------------------------|
| [python/spline_coeff_pack.py](../python/spline_coeff_pack.py)| `MIN_H_CYCLES = 4`           |
| [rtl/spline_eval.sv](../rtl/spline_eval.sv)                  | `parameter MIN_H_CYCLES = 4` |
| [python/autoknots.py](../python/autoknots.py)                | `min_dt` argument            |

## 6.2  `max_dt` — maximum segment width

**Packing origin: 16-bit `h_cycles` field.** The width BRAM word is
32 bits laid out as `[31:16] = recip_h_q115, [15:0] = h_cycles`. The
lower 16-bit `h_cycles` field caps a segment at $2^{16} - 1 = 65535$
cycles, which at 1 GSPS is 65.5 µs.

**Not a physical limit** — it's an artifact of the chosen BRAM word
layout. Widening `h_cycles` in `spline_eval.sv` and reformatting the
width word would lift the cap. Tracked as a future RTL change; for
now, `max_dt` makes sure the algorithm never produces a segment that
wouldn't fit.

**Behavior:** intervals wider than `max_dt` are *force-subdivided*
every iteration, regardless of error condition. Convergence requires
both `n_fail == 0` and `n_wide == 0`. For 1 ms atom-transport, this
turns 13 algorithm-optimal segments into 23 packing-legal ones.

## 6.3  `grid_quantum` — sample-rate knot snapping

**Physical origin: integer cycle counter.** The FPGA's `u_cnt` is
integer cycles. The actual $h_i$ executed by the hardware is
`round(h_s_float * fs)`. If a float spline was built using
$h_s = 6.25\,\mathrm{ns}$ (because bisection produced that midpoint)
but the hardware uses $h_\mathrm{cyc} = 6$, the float reference that
AutoKnots converged against and the float reference the verifier
compares to are **different splines**. This is the bug story
mentioned in Section 5.6.2: adiabatic_ramp showed 44 LSB HW error
from this single source.

**Behavior:** every proposed knot (initial uniform grid, bisection
midpoints, refine-pass midpoints, `seed_knots`) is snapped to a
multiple of `grid_quantum = 1/fs` *before* the local error check
runs. The algorithm's convergence guarantee therefore applies to the
on-grid spline, not a fractional-ns ghost.

> **Why this is the most subtle of the three:** `min_dt` and `max_dt`
> have unambiguous failure signals — a clamp warning or a hard
> ValueError. `grid_quantum`'s failure is a *silently degraded
> waveform*. The tests in 5.6.2 are specifically built so the
> degradation is loud — without `grid_quantum` you get 44 LSB error,
> instantly above the 4 LSB budget.

# 8  Vivado synthesis result — GREEN

This was the missing piece of Phase A. Simulation can prove the
hardware *would* behave correctly if it existed; Vivado synthesis
proves the hardware can actually be built — fits the resources, meets
timing, doesn't trip any structural design-rule checks.

## 8.1  Primer: what synthesis and implementation actually do

> **Primer — the Vivado flow.** Three stages:
>
> 1. **Synthesis** — translate SystemVerilog into a graph of
>    primitives (LUTs, FFs, DSPs, BRAMs).
> 2. **Implementation** — *place* each primitive at a specific
>    location on the silicon, then *route* the wires connecting them.
>    Both placement and routing affect timing because longer routes
>    take longer to propagate signals.
> 3. **Timing analysis** — for every register-to-register path in
>    the design, compute "did the signal arrive at the destination
>    register before the next clock edge?" The answer is reported as
>    *slack*: positive slack = met with margin, negative slack =
>    missed.
>
> *Setup slack* asks "did the signal arrive *before* the clock edge?"
> Positive = good. *Hold slack* asks "did the signal stay valid long
> enough *after* the clock edge?" Positive = good. Both must be
> positive on every path or the design doesn't work.

The two key numbers:

- **WNS** (Worst Negative Slack) — the worst setup slack across all
  paths. If WNS is positive, every path met setup. The closer to
  zero, the tighter the design is to its frequency limit.
- **WHS** (Worst Hold Slack) — same for hold. Usually easy to meet;
  becomes a problem only if minimum delays are too short.

## 8.2  Initial run — GREEN with one YELLOW caveat

The first synthesis pass produced this:

| Metric            | Value             | Interpretation |
|-------------------|-------------------|-----------------|
| **WNS (setup)**   | +0.387 ns         | Met 250 MHz, ~10% of period margin. |
| **WHS (hold)**    | +0.034 ns         | Positive, very tight. |
| Failing endpoints | 0 / 746           | Every path met both setup and hold. |
| **DSPs**          | **7** / 4272 (0.16%) | **Exactly matches design intent**: 6 Horner + 1 $u$-norm. No multiplies fell into LUTs. |
| BRAM tiles        | 0                 | Expected — BRAMs are external to this module. |
| LUTs              | 225 (0.05%)       | Trivial. |
| Flip-flops        | 359 (0.04%)       | Trivial. |
| DRC errors        | 0                 | No structural problems. |
| Critical warnings | 2 (NSTD-1, UCIO-1)| I/O standard and pin-location not set. Not real — they go away on integration into elementconn where pins get assigned. |

> **Why 7 DSPs matters so much.** A DSP slice does a multiply far
> faster and far smaller than an equivalent LUT-based multiplier. If
> Vivado had inferred any of our multiplies as LUT logic instead, the
> design would still work but at much lower frequency and much
> higher resource cost. 7 DSPs = "every multiply we asked for landed
> in a DSP". This is the most important utilization number for a DSP-
> heavy design.

The **YELLOW caveat:** the worst-slack path was a chain of two
back-to-back DSP multiplies in a single clock cycle —
$u_{cnt} \cdot \mathrm{recip\_h}$ (the $u$ normalization) feeding
directly into $d \cdot u$ (Stage 1's first Horner multiply). 0.387 ns
of slack at 250 MHz is real but thin; on a slower-grade part
(`-1` instead of our `-2`), it would likely fail.

## 8.3  The pipelining fix

Defensive engineering: insert one extra flip-flop between the two
DSPs so they each get their own clock period. The cost is one extra
cycle of latency — which we absorb by reducing `OUT_DELAY` from 28 to
27, keeping the 35-cycle total exact.

The change is small. We added **Stage 1a** (a new register stage)
between the BRAM read and the original Stage 1:

```
// Stage 1a: register u_norm_q115 and coeff_data so DSP #1
// (u_cnt*recip_h) and DSP #2 (d*u in Stage 1) are separated by a
// flip-flop. coeff_data has to be delayed in lockstep with
// u_norm_q115 or Stage 1 would see the wrong segment's d/a/b/c for
// the registered u value.
reg signed [15:0] u_norm_q115_reg = 0;
reg [127:0]       coeff_data_reg  = 0;
reg               s1a_gate        = 0;

always @(posedge clk) begin
    if (reset) begin
        u_norm_q115_reg <= '0;
        coeff_data_reg  <= '0;
        s1a_gate        <= 1'b0;
    end else begin
        u_norm_q115_reg <= u_norm_q115;
        coeff_data_reg  <= coeff_data;
        s1a_gate        <= active;
    end
end
```

And updated `OUT_DELAY` from 28 to 27. The latency arithmetic is
explicit in the file's header comment: **8 pipeline cycles + 27
delay = 35 total**.

Why we made *this* specific design choice, not something else:

| Alternative                                  | Why we didn't pick it |
|----------------------------------------------|------------------------|
| Set `PREG = 1` attribute on the DSPs         | Works, but ties the RTL to a vendor-specific synthesis hint. Pipelining via a fabric flop is portable to other tools and grades. |
| Add the register but don't reduce `OUT_DELAY` | Would push the total to 36 cycles, breaking the latency contract with `proc.sv`. |
| Add two register stages for more margin      | Diminishing returns; one stage already nearly doubles the slack. Save complexity for when we actually need it. |
| Replace `q115_mul` function with a `(* use_dsp = "yes" *)` instantiation with all internal pipeline regs | Equivalent result but more verbose, and the function form keeps the math readable. |

math is wrong) and Vivado synthesis. Both passed:

| **WHS (hold)**    | +0.034 ns| +0.032 ns | ~no change |

**Cocotb result: 5/5 pass, error numbers byte-identical to pre-fix.**
This is the most important verification step: we've proven the
latency compensation is correct because the hardware emits exactly
the same samples as before.

**What "Fmax ~307 MHz" means.** Our requirement is 250 MHz (= 4 ns
clock period). We achieved 4 - 0.739 = 3.261 ns of actual delay on
the worst path, which corresponds to a theoretical maximum of
1 / 3.261 ns ≈ 307 MHz. We have 23% margin — enough to absorb
process-voltage-temperature variation, downgrade to a slower silicon
grade, or share the chip with other logic that competes for routing
resources.

**One curiosity worth noting.** The improvement (+0.35 ns) was
smaller than the +1.5 ns I initially projected. The reason is
visible in the post-fix worst-path source: it's
`u_norm_q115_reg_reg/DSP_M_DATA_INST/CLK`. The double-underscore
`_reg_reg` is Vivado's auto-rename and `DSP_M_DATA_INST` indicates
the cell is *inside a DSP*. So Vivado absorbed our new register into
the DSP's internal M-stage (the post-multiplier pipeline register)
rather than placing it as a separate fabric flop. The DSP got better,
but the path is still DSP-output → DSP-input. To push slack further,
we'd need an *additional* register that Vivado can't absorb — a
follow-up optimization, not necessary for the current 250 MHz target.

# 9  What's left after this — Phase B integration

Phase A scope ends at "the RTL synthesizes cleanly, meets timing,
matches the simulator byte-for-byte." Phase B is the integration work:

1. **Drop `spline_eval` into `elementconn.v`.** Replace the envelope
   BRAM read path. The `env_word` bit mapping (`[23:12]=coeff_base,
   [11:0]=n_segs`) is unchanged from the original, so `pulse_reg.sv`,
   `pulse_iface.sv`, and `proc.sv` need *no* edits. The 35-cycle
   contract guarantees this.

2. **Instantiate two Vivado BMG IPs** for the coefficient and width
   BRAMs, initialized from the `.coe` files this project produces.
   Configure as simple-dual-port, read latency 2, with the second
   port runtime-writable for calibration updates.

3. **Hardware regression.** Fire actual gates on a real qubit, compare
   measured Rabi/readout outcomes to the existing raw-sample path.
   This is the final gate before the new path replaces the old one in
   production firmware.

The RTL contract documented here is intended to be sufficient for
that work — anyone who knows QubiC should be able to do the
integration without needing to re-derive the Q1.15 conventions, the
prefetch logic, or the latency padding.

# 10  Decisions, distilled

A summary of what we chose, why, and where it shows up in the code.
This is the list to skim if you're trying to understand "why does
the project look the way it does?"

| Decision                                           | Why                                                                                          | Where to look |
|----------------------------------------------------|-----------------------------------------------------------------------------------------------|---------------|
| Cubic splines, not Bezier / Chebyshev / linear     | $C^2$ + local + cheap. Sweet spot for "smooth enough that DAC can't tell."                   | [autoknots.py](../python/autoknots.py) |
| Horner form, not direct polynomial                 | 3 multiplies + 3 adds instead of 6+3. Halves DSP count.                                      | [spline_eval.sv:235-363](../rtl/spline_eval.sv#L235) |
| Q1.15 fixed-point, not float, not Q0.16            | Matches DAC bit-depth exactly. Fits cleanly in DSP48E2. Range covers QubiC's $\mathrm{amp} < 1$. | [spline_coeff_pack.py](../python/spline_coeff_pack.py) |
| Pre-scale coefficients by $h_s^k$ in Python        | Hardware sees $u \in [0,1)$; no divider needed; coefficients fit Q1.15.                       | [spline_coeff_pack.py:198-208](../python/spline_coeff_pack.py#L198) |
| `recip_h = round(32768/h)` precomputed in Python   | Replaces hardware division with one multiply.                                                | [spline_coeff_pack.py:196](../python/spline_coeff_pack.py#L196) |
| Latency padding via `reg_delay1 OUT_DELAY`         | Preserves 35-cycle contract → zero edits to other QubiC files.                               | [spline_eval.sv:365-377](../rtl/spline_eval.sv#L365) |
| External BRAMs, not inside `spline_eval`           | Matches existing QubiC envelope-memory structure; runtime calibration writes still work.     | port interface, [spline_eval.sv:108-115](../rtl/spline_eval.sv#L108) |
| Saturating output stage                            | Spline overshoot near peak amp wraps without it → DAC glitches.                              | [spline_eval.sv:334-351](../rtl/spline_eval.sv#L334) |
| Sticky prefetch register                           | Single-cycle compare would refire on the next cycle and refetch wrong data.                  | [spline_eval.sv:184-213](../rtl/spline_eval.sv#L184) |

## Appendix B — Glossary

| Term                | Meaning |
|---------------------|---------|
| **AutoKnots**       | The adaptive knot-allocation algorithm we use (Vitenti et al. 2024). |
| **BRAM**            | Block RAM. On-chip memory tile on the FPGA. We use Vivado BMG IP for the two we need. |
| **BRAM tile**       | The physical unit of BRAM on the chip (36 kbit). ZCU216 has 1080. |
| **`busy_out`**      | A status output of `spline_eval` that says "I'm currently playing a pulse." |
| **CLB**             | Configurable Logic Block. The repeatable unit of FPGA fabric containing LUTs and FFs. |
| **cocotb**          | A Python library for HDL testbenches. Lets us drive SystemVerilog from Python. |
| **`cmdstb`**        | "Command strobe." A one-cycle pulse signaling "start a new gate now." |
| **DAC LSB**         | Least significant bit of the 16-bit DAC. The fundamental error unit in this project. |
| **DRAG**            | Derivative Removal by Adiabatic Gate. A pulse shape that suppresses leakage to $|f\rangle$. |
| **DRC**             | Design Rule Check. Vivado's structural sanity check on the routed design. |
| **DSP48E2**         | The DSP slice on UltraScale+ FPGAs. Signed 27×18 multiply + accumulate. |
| **`envdata_out`**   | The output of `spline_eval`: 32 bits packed as `{I[15:0], Q[15:0]}`. |
| **Fmax**            | Maximum clock frequency the design could be run at and still meet timing. |
| **FF**              | Flip-flop. A single-bit clocked storage element. |
| **`grid_quantum`**  | The time-axis quantum (1/fs = 1 ns at 1 GSPS). All knots must land on a multiple of this. |
| **Horner form**     | Nested polynomial evaluation: $a + u(b + u(c + ud))$. Half the multiplies of the naive form. |
| **Icarus Verilog**  | The open-source SystemVerilog simulator we use for cocotb tests. |
| **knot**            | A breakpoint in a spline; the segments between knots each have their own cubic. |
| **LUT**             | Look-up table. Programmable boolean logic. A 6-LUT computes any function of 6 inputs. |
| **`max_dt`**        | Maximum segment width. Set by the 16-bit `h_cycles` BRAM field. |
| **`min_dt`**        | Minimum segment width. Set by the prefetch FSM's correctness requirement. |
| **opt_design / place / route** | The three implementation stages in Vivado after synthesis. |
| **Q1.15**           | Signed fixed-point: 1 sign bit + 0 integer bits + 15 fraction bits. Range $[-1, +1)$. |
| **QubiC**           | The LBNL superconducting-qubit control system this project plugs into. |
| **`recip_h`**       | Precomputed reciprocal of `h_cycles` in Q1.15. Avoids a hardware divider. |
| **RFSoC**           | "RF System on Chip." Xilinx product family combining FPGA fabric with high-speed RF DAC/ADC. |
| **`seg_idx`**       | Current segment number within a playing pulse. |
| **slack**           | (Required time) − (arrival time) on a timing path. Positive = met. |
| **SRL**             | Shift Register LUT. A LUT used as a small shift register; very efficient. |
| **synthesis**       | Translating HDL into a graph of primitives. First Vivado stage. |
| **timing closure**  | Achieving WNS ≥ 0 and WHS ≥ 0. The basic correctness gate of synthesis. |
| **`u_cnt`**         | Counter for intra-segment cycles. Goes from 0 to `h_cycles - 1`. |
| **`u_norm`**        | Normalized intra-segment time, in $[0, 1)$. The variable in the Horner polynomial. |
| **`valid_out`**     | A signal that says "the value on `envdata_out` this cycle is a real sample." |
| **WHS**             | Worst Hold Slack. Hold timing margin on the worst path. |
| **WNS**             | Worst Negative Slack. Setup timing margin on the worst path. |
| **XDC**             | Xilinx Design Constraints. The file format for clock and pin constraints. |
| **ZCU216**          | The Xilinx RFSoC evaluation board QubiC runs on. Part: `xczu48dr-ffvg1517-2-e`. |
